### 2. extra_body 修复
- Codex CLI 通过 `extra_body` 传递 thinking/reasoning_effort 等参数
- DeepSeek 不认识 `extra_body`，需展开到请求体顶级
- 代码已更新：extra_body 内容逐字段展开，不覆盖已有参数

### 3. reasoning_content 回传修复
- DeepSeek 要求：多轮对话有工具调用时必须回传 reasoning_content
- Codex 回传的是 reasoning item（含 encrypted_content），需提取 summary
- 添加 `extractReasoningSummary` 函数和 `pendingReasoningContent` 机制

### 4. 流式转换器 status 修复
- 有工具调用时 `status` 应为 `incomplete`，不是 `completed`
- 添加 `finalStatus` 和 `incompleteDetails` 逻辑

### 5. response_format 400 报错修复
- DeepSeek 只支持 `text` 和 `json_object`，不支持 `json_schema`
- Codex CLI 发 `text.format.type: "json_schema"` 导致 400
- 修复：思考模式下跳过 response_format，非思考模式 json_schema 降级为 json_object

### 6. 流式转换器 ID 一致性修复
- `processChunk` 中为 reasoning/message item 生成的 ID 必须保存
- `finalize` 中 `response.output_item.done` 和 `response.completed` 的 output 数组必须使用相同 ID
- 之前每次 `genId('rs')` 都生成新 ID，导致 Codex CLI 无法匹配 item

### 7. 去掉 data: [DONE]
- Responses API 流以 `response.completed` 事件终止
- `data: [DONE]` 是 Chat Completions 的格式，不应该出现在 Responses API 流中
- Codex CLI 收到意外的 `data: [DONE]` 可能干扰解析

### 8. 防止双重 res.end()
- `upstreamRes.on('data')` 检测到 `[DONE]` 和 `upstreamRes.on('end')` 都会触发 `endStream()`
- 添加 `streamEnded` 标志防止双重调用

### 9. 流式请求上游错误的 SSE 格式
- 上游返回 400/500 时，不能直接转发 JSON
- Codex CLI 期望 SSE 格式，收到纯 JSON 会认为流断开
- 改为以 SSE 格式返回 `response.created` + `response.completed`（status=failed）

## 踩坑经验
- DeepSeek response_format 不支持 json_schema → 必须降级或跳过
- 思考模式下 response_format 兼容性更差 → 建议跳过
- **`data: [DONE]` 是必须的！** OpenAI SDK 用它作为流终止信号
  - 不发 [DONE] → SDK 认为流异常断开 → "stream disconnected before completion"
  - 之前误以为 Responses API 不需要 [DONE]，删除后导致断流
- SSE 事件中的 item ID 必须在整个流中保持一致
- 有 tool_calls 但没有文本内容时，不创建空的 message item
